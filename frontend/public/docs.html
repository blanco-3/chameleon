1. Abstract

Chameleon is a privacy-preserving payment pool built on the Stellar blockchain that allows users to deposit and withdraw fixed-denomination XLM without revealing the link between depositor and recipient — while simultaneously enforcing a compliance blacklist that cryptographically prevents sanctioned addresses from withdrawing.

The protocol combines a Poseidon2-based Merkle accumulator with an UltraHonk zero-knowledge proof system (Noir + Barretenberg) and an on-chain verifier deployed as a Soroban smart contract. A depositor commits a secret note on-chain; a withdrawer later proves membership in the commitment set and non-membership in a compliance blacklist, without disclosing which deposit they control.

Key Property Privacy and compliance are not in tension in Chameleon — they are enforced simultaneously by the same zero-knowledge proof. A sanctioned user cannot generate a valid proof even offline.

2. Introduction & Motivation

2.1 The Problem with Transparent Blockchains

Every transaction on Stellar is permanently public. When address A sends 100 XLM to address B, this link is recorded on-chain and visible to anyone forever. This creates significant privacy challenges: businesses reveal counterparties, individuals expose their financial history, and even pseudonymous addresses can be de-anonymized through chain analysis.

2.2 Existing Approaches and Their Shortcomings

Tornado Cash demonstrated that zero-knowledge proofs can break the transaction graph on EVM chains. However, it has two critical problems for real-world adoption:

3. Cryptographic Primitives

3.1 Poseidon2 Hash Function

All hashing in Chameleon uses Poseidon2, an algebraic hash function optimized for zero-knowledge proof systems. Unlike SHA-256 or Keccak, Poseidon2 operates natively over finite field elements, making its constraints inside arithmetic circuits orders of magnitude cheaper.

We use the BN254 scalar field (order ≈ 2²⁵⁴) with a width-4 (t=4) permutation and a sponge construction with rate RATE=3:

hash₂(a, b) = Poseidon2_perm([a, b, 0, 2·2⁶⁴])[0] hash₁(a) = Poseidon2_perm([a, 0, 0, 1·2⁶⁴])[0] IV = input_length × 2⁶⁴ (Noir's Poseidon2Hasher convention)

The IV encodes the input length, preventing length-extension attacks. This construction is identical to Noir's built-in std::hash::poseidon2 and has been verified to produce bit-identical outputs across three independent implementations: Noir circuit, TypeScript CLI, and Rust contract tests.

3.2 Merkle Tree

The commitment accumulator is a binary Merkle tree of depth 20, supporting up to 2²⁰ ≈ 1,048,576 deposits. Each leaf stores a commitment; internal nodes are Poseidon2 hashes of their two children. Empty subtrees use a precomputed zero-value chain:

ZERO[0] = 0 ZERO[i] = Poseidon2(ZERO[i-1], ZERO[i-1]) depth = 20 capacity = 2²⁰ = 1,048,576 deposits

The Merkle root summarizes all deposits and changes with every new insertion. To mitigate race conditions between concurrent deposits and proof generation, the contract stores the last 30 valid roots. A proof generated against any of these roots is accepted.

3.3 UltraHonk Proof System

Chameleon uses UltraHonk, a recent proof system from the Aztec/Barretenberg project with the following properties relevant to Soroban deployment:

4. Protocol Design

4.1 Note Generation

A note is a pair (nullifier, secret) of cryptographically random 256-bit field elements. The user must store this note securely — it is the only credential needed to withdraw. Loss of the note is permanent loss of funds.

nullifier ← random(𝔽_BN254) secret ← random(𝔽_BN254) commitment = Poseidon2(nullifier, secret)

4.2 Deposit

The depositor calls deposit(from, commitment) on the contract, transferring exactly DENOMINATION (100 XLM = 10⁹ stroops) from their account. The contract inserts the commitment into the next leaf of the Merkle tree and emits a Deposited event containing the leaf index and commitment.

Only the commitment is recorded on-chain. The nullifier and secret never leave the user's device.

4.3 Sync

Before proving, the user must reconstruct the current Merkle tree by fetching all Deposited events from the contract and reinserting commitments in order. The CLI caches this state locally. The root of the reconstructed tree must match one of the last 30 roots stored on-chain.

4.4 Proof Generation

The CLI writes a Prover.toml containing all public and private inputs, then invokes nargo execute (witness generation) followed by bb prove --scheme ultra_honk --oracle_hash keccak (proof generation). The resulting proof.json contains:

FieldSizeDescription
proof14,592 BRaw UltraHonk proof bytes (hex)
publicInputs192 B6 × 32-byte field elements (hex)
humanReadable—Named public inputs for debugging

4.5 Withdrawal

The withdrawer submits the proof to the contract's withdraw function. The contract verifies:

  1. The Merkle root in the proof is a known recent root
  2. The nullifier hash has not been spent before
  3. The blacklist root in the proof matches the current on-chain root
  4. The UltraHonk ZK proof is valid (on-chain verification)

If all checks pass, the nullifier hash is recorded and 100 XLM is transferred to the recipient. The recipient address is bound into the proof, preventing front-running.

The Chameleon circuit is written in Noir and compiled to an ACIR (Abstract Circuit Intermediate Representation) artifact. It enforces six constraints:

5.2 Circuit Constraints

// Constraint 1: commitment is correctly formed
let commitment = hash_2(nullifier, secret);

// Constraint 2: nullifier_hash is correctly formed  
assert(hash_1(nullifier) == nullifier_hash);

// Constraint 3: commitment exists in the Merkle tree
let computed_root = merkle_root(commitment, path_elements, path_indices);
assert(computed_root == root);

// Constraint 4: commitment is not in the compliance blacklist
assert_not_blacklisted(commitment, blacklist);

// Constraint 5: public inputs are anchored (anti-malleability)
let _anchor = recipient + relayer + fee + blacklist_root;

5.3 Merkle Verification

The Merkle path verification recomputes the root from the leaf commitment and 20 sibling hashes. At each level, the hasher chooses left/right order based on the path index bit: 

fn merkle_root(leaf: Field, siblings: [Field; 20], indices: [Field; 20]) -> Field {
    let mut current = leaf;
    for i in 0..20 {
        let (left, right) = if indices[i] == 0 {
            (current, siblings[i])
        } else {
            (siblings[i], current)
        };
        current = hash_2(left, right);
    }
    current
}

5.4 Non-Membership Proof

The blacklist check is an O(k) linear scan over 16 slots. Zero-valued slots are treated as empty sentinels and skipped. If any non-zero slot equals the withdrawer's commitment, the constraint fails and no valid proof can be generated: 

fn assert_not_blacklisted(commitment: Field, blacklist: [Field; 16]) {
    for i in 0..16 {
        if blacklist[i] != 0 {
            assert(commitment != blacklist[i]);
        }
    }
}
Security Property The non-membership check runs inside the ZK circuit. A sanctioned user cannot bypass it by submitting a false proof — the proof system guarantees the constraint was satisfied. The only way to forge a proof would be to break the discrete-log hardness assumption underlying BN254.

6.3 On-Chain Proof Verification

The rs-soroban-ultrahonk crate implements UltraHonk verification natively in Soroban, without any trusted setup. The verification key (1,760 bytes) is stored on-chain at initialization and used to verify every withdrawal proof: 

let vk = get_vk(&env)?;
if !verifier::verify(&env, &vk, &proof, &public_inputs) {
    return Err(PoolError::InvalidProof);
}

The verifier operates on 14,592-byte proof blobs and 192-byte public input blocks (6 × 32-byte big-endian field elements).

7. Compliance Mechanism

7.1 Design Philosophy

Chameleon's compliance system is inclusion-proof-based: instead of reactively blacklisting withdrawals, it requires every withdrawer to prove they are not sanctioned. This is enforced at proof-generation time by the ZK circuit — not merely at contract validation time.

7.2 Blacklist Lifecycle

Admin Contract │ │ │── set_blacklist(root, entries) ───▶│ store BlacklistRoot = root │ │ store BlacklistEntries[c] = true │ │ for each entry c in entries │ │ (root is public; entries are indexed for contract-level checks) │ Withdrawer (sanctioned): │ │ Prover.toml includes blacklist = [sanctioned_commitment, ...] │ nargo execute → FAIL: assert(commitment != entry) violated │ → Cannot generate proof → Cannot withdraw

7.3 Freshness Enforcement

The blacklist_root is a public input bound into every ZK proof. The contract stores the current root and rejects any proof whose blacklist_root does not match (StaleBlacklist).

This means: when the admin updates the blacklist, all previously generated proofs are immediately invalidated. A sanctioned user cannot use a proof generated before their address was added to the list.

7.4 Hackathon Scope Note

Current Limitation The blacklist_root is not cryptographically derived from the entries inside the ZK circuit — it is an externally provided commitment. In a production system, the circuit would compute blacklist_root = Poseidon2_tree(entries) and verify this computation, creating a cryptographic binding. The current design relies on the admin providing consistent root/entries pairs and on the contract's entry registry for double-checking. Production-grade non-membership proofs would use an indexed Merkle tree (e.g., Semaphore-style) to make the binding explicit inside the circuit.

8. Security Analysis

8.1 Sender-Receiver Unlinkability

Given a withdrawal proof, an adversary observing the blockchain sees:

  • The nullifier_hash (a hash of the nullifier, not linkable to the commitment)
  • The recipient address
  • The Merkle root (summarizes all deposits, not which one is spent)

Linking a specific withdrawal to a specific deposit requires inverting commitment = Poseidon2(nullifier, secret) and nullifier_hash = Poseidon2(nullifier) simultaneously, which is computationally infeasible under standard hash function assumptions. The anonymity set is the number of deposits made since the last sync.

8.2 Double-Spend Prevention

Each withdrawal produces a unique nullifier_hash derived from the note's nullifier. The contract records every spent nullifier hash. A second withdrawal attempt with the same note returns NullifierSpent before reaching the proof verifier.

8.3 Front-Running Prevention

The recipient, relayer, and fee fields are bound into the ZK proof as public inputs. A front-runner who intercepts a withdrawal transaction cannot redirect funds to a different address without invalidating the proof.

8.4 Proof Soundness

UltraHonk is a knowledge-sound proof system under the discrete-log hardness assumption on BN254. Forging a proof for a commitment not in the Merkle tree, or bypassing the non-membership check, would require computing a discrete log over a 254-bit curve — infeasible for any classical adversary.

11. Conclusion

Chameleon demonstrates that privacy and compliance are not mutually exclusive on a transparent blockchain. By embedding a non-membership proof inside the withdrawal circuit, the protocol achieves strong sender-receiver unlinkability while guaranteeing that sanctioned addresses cannot withdraw — without requiring the compliance system to learn anything about legitimate users.

The on-chain UltraHonk verifier (rs-soroban-ultrahonk) enables trustless, permissionless verification of 14,592-byte proofs inside Soroban smart contracts — a novel capability on the Stellar network. All code is open-source and the contract is deployed on Stellar Testnet.
